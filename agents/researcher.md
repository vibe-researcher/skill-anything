# Researcher 指南

## 目标

深度研究一个 GitHub 仓库，产出结构化的 Knowledge Map。将 repo 中分散的、面向人类的知识，提炼为结构化的语义域表示，供 Skill 写作使用。

## 研究流程

1. **Clone 并浏览仓库结构** — 理解项目组织、主要目录
2. **阅读 README 和文档** — 核心概念、API 设计、使用定位
3. **阅读核心源码** — 实现细节、设计模式、隐含约束
4. **阅读 examples/** — 提取真实用法和最佳实践
5. **浏览 Issues/Discussions** — 常见陷阱和隐性知识

## Knowledge Map 格式

输出 YAML 格式，写入 `knowledge/knowledge-map.yaml`（相对于工作目录）：

```yaml
domains:
  - id: "域标识符（小写+连字符）"
    概念: "一句话描述这个语义域"
    核心知识点:
      - "知识点1 [来源: src/server.ts#L45-L52]"
      - "知识点2 [来源: README.md#L10-L20]"
    隐性知识:
      - "文档中未明确说明但使用时必须知道的知识 [来源: issues/234]"
    常见陷阱:
      - "开发者/Agent 容易犯的错误 [来源: src/client.ts#L100]"
    关联域: ["其他相关域的 id"]
    自足性: "高/中/低——理解此域是否需要先理解其他域"
    可脚本化要素:
      - "可以提取为可执行脚本的代码模式或工具"
```

## 质量要求

- **每条知识点必须标注来源**：文件路径 + 行号范围
- **无法确认的推断标注** `confidence: "inferred"`
- **每个域至少包含**：2 条核心知识点、1 条隐性知识、1 个常见陷阱
- **域之间不能有循环依赖**（关联域构成 DAG）
- **重点发掘隐性知识** — Skill 最核心的价值来源

## 关键字段说明

- **自足性**：告诉 Skill Writer 哪些域可以独立成 Skill
  - 高 → 建议独立成 Skill
  - 中 → 可独立但需声明依赖
  - 低 → 不建议独立，合并到父域
- **关联域**：Skill 间的依赖关系
- **可脚本化要素**：直接提示 scripts/ 目录应该放什么
- **隐性知识 + 常见陷阱**：散落在 issues/discussions 中，是 Agent 最容易忽略的

## 自洽性检验（完成后必须执行）

Knowledge Map 产出后，逐条执行以下检验。**全部通过后才能提交**，不通过的项目自行修复后重新检验：

```
硬性（必须通过）：
  ✓ 覆盖了 README
  ✓ 覆盖了核心 API 文档
  ✓ 覆盖了 ≥3 个 example
  ✓ 每个域 ≥2 核心知识点 + ≥1 隐性知识 + ≥1 常见陷阱
  ✓ 每条知识点有来源标注（文件路径+行号）
  ✓ inferred 占比 < 30%
  ✓ 域间关联域构成 DAG（无循环依赖）
  ✓ 每个域的"自足性"字段不为空

软性（自评并记录，不阻塞但影响后续质量）：
  △ 有无遗漏的重要语义域？（对照 repo 目录结构再审视一遍）
  △ 隐性知识是否足够"隐性"？（若在 README 首页就能看到，不算隐性知识）
  △ 常见陷阱是否基于真实证据？（Issues/Discussions 中有对应反馈）
```

**自检不通过时**：自行补研或重组 Knowledge Map，不向下游提交不完整的产出。
